# feat: Phase 14 — Assessment System (AI pre-screening before first human contact)

> **Labels:** `feature`, `phase-14`
> **AI Evaluator:** Gemini 2.0 Flash via `generativelanguage.googleapis.com`
> **Feature flag:** `ASSESSMENT_SYSTEM_ENABLED` (default: `false`)
> **Full design doc:** `docs/assessment-system-design.md`

---

## 1. Цель

Реализовать систему автоматического отбора кандидатов **до первого живого интервью** для двух ролей: `logist` (Логист-экспедитор) и `sales_manager` (Менеджер по продажам ТЭУ). Кандидат проходит 4 этапа строго последовательно — анкету, профессиональный тест с жёстким таймером, психологический тест и практическое тестовое задание. После завершения всех этапов BullMQ-очередь запускает AI-оценщик на базе **Gemini 2.0 Flash**, который анализирует все ответы, выполняет кросс-проверку на признаки лжи и выносит один из трёх вердиктов: **ДОПУСТИТЬ / ОТКЛОНИТЬ / НА РУЧНУЮ ПРОВЕРКУ HR**. Это позволяет HR-команде тратить время только на кандидатов, прошедших автоматическую проверку, и получать структурированный AI-отчёт с флагами рисков ещё до первого звонка.

---

## 2. Архитектура

- **AI-оценщик:** Gemini 2.0 Flash via `fetch` на endpoint `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent`, ключ из env `GEMINI_API_KEY`
- **Queue:** BullMQ job `assessment:evaluate` — запускается автоматически после успешной отправки Этапа 4
- **Публичная страница кандидата:** token-based URL (авторизация не требуется), строгий серверный таймер для Этапа 2 (проверка `submitted_at - started_at <= 30min`)
- **Feature flag:** `ASSESSMENT_SYSTEM_ENABLED` (default `false` в prod)

## 3. Схема данных

### Новые таблицы

```sql
-- Шаблон оценочной сессии (привязан к вакансии)
CREATE TABLE assessment_templates (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  vacancy_id UUID REFERENCES vacancies(id),
  role TEXT NOT NULL CHECK (role IN ('logist', 'sales_manager')),
  stages JSONB NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- Сессия прохождения кандидатом
CREATE TABLE assessment_sessions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  application_id UUID REFERENCES applications(id),
  template_id UUID REFERENCES assessment_templates(id),
  status TEXT NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending','stage_1','stage_2','stage_3','stage_4','completed','rejected','expired')),
  started_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ,
  expires_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- Ответы по каждому этапу
CREATE TABLE assessment_stage_results (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID REFERENCES assessment_sessions(id),
  stage_number INT NOT NULL CHECK (stage_number IN (1,2,3,4)),
  answers JSONB NOT NULL,
  scores JSONB,
  flags JSONB,
  ai_evaluation JSONB,
  submitted_at TIMESTAMPTZ DEFAULT now()
);

-- Итоговый вердикт AI
CREATE TABLE assessment_verdicts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID REFERENCES assessment_sessions(id) UNIQUE,
  verdict TEXT NOT NULL CHECK (verdict IN ('ДОПУСТИТЬ','ОТКЛОНИТЬ','НА РУЧНУЮ ПРОВЕРКУ HR')),
  total_weighted_score NUMERIC(5,2),
  stage_scores JSONB,
  cross_check_flags JSONB,
  lie_scale_result JSONB,
  verdict_reason TEXT,
  hr_notes TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

### RLS политики
- Кандидат видит только свою сессию по токену (без авторизации в системе)
- HR видит все сессии по своим вакансиям
- AI-сервис пишет через service_role

## 4. FSM сессии

```
pending
  └─→ stage_1 (анкета открыта)
        └─→ stage_1_rejected (стоп-критерий сработал) → КОНЕЦ
        └─→ stage_2 (анкета пройдена)
              └─→ stage_2_failed (тест < порога) → КОНЕЦ
              └─→ stage_3
                    └─→ stage_4
                          └─→ completed (все этапы сданы)
                                └─→ [AI verdict]
```

Переход между этапами — максимум **24 часа**. Просрочка → сессия `expired`.

## 5. Этапы

| Этап | Название | Время | Тип оценки |
|------|----------|-------|------------|
| 1 | Анкета-скрининг | Нет лимита | Автоматически (стоп-критерии) |
| 2 | Профессиональный тест | 30 мин строго | AI + автопроверка |
| 3 | Психологический тест | Нет лимита | AI |
| 4 | Тестовое задание | 35–45 мин | AI (рубрика) |

## 6. Механизм антилжи (cross-check)

| # | Тип | Что проверяем |
|---|-----|---------------|
| 1 | 🔴 RED | Ловушка: несуществующая TMS / методология — автоотказ |
| 2 | 🔴 RED | Несуществующий конкурент в Анкете — автоотказ |
| 3 | 🔴 RED | Заявленный опыт не подтверждается тестом |
| 4 | 🔴 RED | Нет документов при заявленном международном опыте |
| 5 | 🟠 ORANGE | L-шкала психотеста: 3+ ответа «5» |
| 6 | 🟠 ORANGE | Устаревший Инкотермс FOR заявлен как рабочий |
| 7 | 🟠 ORANGE | Нештатка в анкете детальная, в тесте — примитивная |
| 8 | 🟠 ORANGE | Неравномерность частей тестового задания |

**Правило:** 2+ RED = автоотказ. 1 RED или 1–3 ORANGE = НА РУЧНУЮ ПРОВЕРКУ HR.

## 7. Весовая формула

**Веса этапов:**
- Этап 1: бинарный фильтр
- Этап 2: **40%**
- Этап 3: **20%**
- Этап 4: **40%**

**Формула:** `score = (E2/35)*40 + (E3/64)*20 + (E4/role_max)*40`

Где `role_max` = 25 для логиста, 23 для менеджера продаж.

**Вердикт — строго JSON:**
```typescript
interface AssessmentVerdict {
  candidate_id: string;
  role: string;
  verdict: 'ДОПУСТИТЬ' | 'ОТКЛОНИТЬ' | 'НА РУЧНУЮ ПРОВЕРКУ HR';
  total_weighted_score: number; // 0–100
  stage_scores: { stage_2_score: number; stage_3_score: number; stage_4_score: number };
  cross_check_flags: { flag_id: number; type: 'RED' | 'ORANGE'; description: string; impact: string }[];
  lie_scale_result: { score_5_count: number; reliability: 'RELIABLE' | 'MODERATE_RISK' | 'UNRELIABLE' };
  verdict_reason: string;
  hr_notes: string;
}
```

## 9. AI-промпт оценщика

### Системный промпт для AI-оценщика (Onboardix Assessment Engine)

Вызывать через `fetch` на:
```
POST https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent?key=${GEMINI_API_KEY}
```

Системный промпт (передавать как `system_instruction.parts[0].text`):

```
Ты — AI-оценщик системы подбора персонала Onboardix. Твоя задача: принять данные кандидата по всем 4 этапам отбора и выдать ЕДИНЫЙ ВЕРДИКТ по одной из трёх позиций: ДОПУСТИТЬ / ОТКЛОНИТЬ / НА РУЧНУЮ ПРОВЕРКУ HR.

---

## ВХОДНЫЕ ДАННЫЕ

Ты получаешь структурированный JSON с данными кандидата:
- candidate_id: идентификатор
- role: "logist" | "sales_manager"
- stage_1: данные анкеты (ответы, флаги стоп-критериев, флаги ловушек)
- stage_2: результаты профессионального теста (баллы по вопросам, открытые ответы)
- stage_3: результаты психологического теста (баллы по блокам, L-шкала)
- stage_4: тестовое задание (текст ответа кандидата)
- cross_check_flags: массив флагов несоответствий, выявленных на предыдущих этапах

---

## ВЕСА ЭТАПОВ

| Этап | Вес в итоговой оценке |
|------|-----------------------|
| Этап 1 (Анкета) | Бинарный фильтр — при стоп-критерии: ОТКЛОНИТЬ без дальнейшего анализа |
| Этап 2 (Профтест) | 40% итоговой оценки |
| Этап 3 (Психотест) | 20% итоговой оценки |
| Этап 4 (Тестовое задание) | 40% итоговой оценки |

Итоговый балл = (Этап2_балл / Этап2_макс) × 40 + (Этап3_балл / Этап3_макс) × 20 + (Этап4_балл / Этап4_макс) × 40

---

## ПРАВИЛА ВЫНЕСЕНИЯ ВЕРДИКТА

### ОТКЛОНИТЬ — автоматически, если выполняется ХОТЯ БЫ ОДНО:
1. Сработал стоп-критерий на Этапе 1 (зарплата, локация, опыт ниже минимума, формат работы).
2. Кандидат подтвердил знание несуществующего продукта/компании (ловушки анкеты) — КРАСНЫЙ флаг.
3. Итоговый балл Этапа 2 ниже порога (< 22/35 для обеих ролей).
4. Итоговый балл Этапа 4 ниже порога (< 16/25 для логиста, < 15/23 для менеджера продаж).
5. Итоговый взвешенный балл < 50%.
6. Сработало 2 и более КРАСНЫХ флага cross-check.

### ДОПУСТИТЬ — если выполняются ВСЕ условия:
1. Ни одного стоп-критерия на Этапе 1.
2. Ни одного КРАСНОГО флага cross-check.
3. Этап 2 ≥ 22 балла.
4. Этап 4 ≥ порог роли.
5. Итоговый взвешенный балл ≥ 65%.
6. L-шкала: не более 1 ответа «5».

### НА РУЧНУЮ ПРОВЕРКУ HR — во всех остальных случаях, включая:
- 1 КРАСНЫЙ флаг без автоотказа (например, ловушка в анкете при пограничном тесте).
- 1–3 ОРАНЖЕВЫХ флага.
- Итоговый балл 50–64%.
- L-шкала: 2 ответа «5» (умеренный риск).
- Неравномерность результатов: один этап значительно выше другого.
- Любая неоднозначная ситуация, не покрытая правилами выше.

---

## ИНСТРУКЦИЯ ПО АНАЛИЗУ ОТКРЫТЫХ ОТВЕТОВ

При оценке открытых ответов (Этапы 2 и 4) применяй следующие критерии:

1. **Конкретность:** есть ли реальные детали (цифры, названия, последовательность действий) или только общие слова?
2. **Профессиональная логика:** соответствуют ли действия кандидата отраслевой практике? Есть ли логические ошибки?
3. **Соответствие заявленному опыту:** если кандидат заявил 5 лет опыта — ответ должен быть нетривиальным. Ответы уровня «начинающего» при большом стаже — флаг.
4. **Работа с ловушками:** заметил ли кандидат намеренные противоречия или ошибки в задании? Отметил ли недостающие данные?
5. **Честность vs исполнение:** кандидат честно признаёт ограничения («не хватает данных») или делает вид, что всё понял?

---

## ИНСТРУКЦИЯ ПО ФЛАГАМ CROSS-CHECK

Получи массив cross_check_flags. Для каждого флага:
- Тип КРАСНЫЙ: обязательно упомянуть в вердикте, применить правила автоотказа.
- Тип ОРАНЖЕВЫЙ: описать несоответствие, снизить доверие к соответствующим этапам на 20%, передать HR для уточнения.
- Если флагов нет — зафиксировать как «Профиль без выявленных несоответствий».

- Ты НЕ даёшь расплывчатых ответов. Вердикт всегда один из трёх.
- Ты НЕ снижаешь стандарты из соображений «дефицита кандидатов». Планка фиксирована.

- Открытые ответы оцениваются по рубрике, но финальное суждение об их качестве — твоё аналитическое решение. Обоснуй его одним предложением.
```

## 10. Контент вопросов

---

### РОЛЬ: ЛОГИСТ-ЭКСПЕДИТОР

---

#### ЭТАП 1: АНКЕТА-СКРИНИНГ — Логист-экспедитор

##### Вопрос 1. Ожидаемый уровень дохода

**Текст:** Укажите ваши ожидания по зарплате (фикс, на руки, в рублях). Если ожидания состоят из фикса и бонуса — укажите оба.

**Тип:** Открытый (числовой)

**Стоп-критерий:** Фикс выше установленного вилки найма (настраивается под вакансию, например > 120 000 руб.). Автоматический отказ с формулировкой: «Уровень дохода выходит за рамки бюджета позиции».

---

##### Вопрос 2. Общий опыт в логистике и экспедировании

**Текст:** Сколько лет вы работаете в сфере транспортной логистики и/или экспедирования? Включите весь опыт, в том числе частичную занятость.

**Тип:** Одиночный выбор

**Варианты:**
- Менее 1 года
- 1–2 года
- 3–5 лет
- Более 5 лет

**Стоп-критерий:** «Менее 1 года» — автоматический отказ, если в вакансии минимальный порог 1 год. Сохраняется как флаг для cross-check с тестом.

---

##### Вопрос 3. Форматы перевозок, с которыми работали

**Текст:** Отметьте все направления, с которыми вы имеете практический опыт работы (не просто знаете теоретически).

**Тип:** Множественный выбор

**Варианты:**
- Автомобильные перевозки (FTL/LTL)
- Железнодорожные перевозки
- Авиаперевозки
- Морские перевозки (FCL/LCL)
- Мультимодальные перевозки
- Таможенное оформление (декларирование)
- Международные перевозки (страны СНГ)
- Международные перевозки (дальнее зарубежье)

**Стоп-критерий:** Если вакансия требует конкретного формата (например, международные перевозки ж/д) — отсутствие соответствующего пункта автоматически переводит в статус «На проверку HR», не автоотказ. Сохраняется для cross-check.

---

##### Вопрос 4. Локация и готовность к работе в офисе

**Текст:** Укажите ваш город проживания и готовность к работе в офисе (г. [ГОРОД]).

**Тип:** Текст + одиночный выбор

**Варианты готовности:**
- Готов работать в офисе 5/5
- Готов на гибридный формат (2–3 дня в офисе)
- Только удалённо

**Стоп-критерий:** «Только удалённо», если позиция требует присутствия. Другой город без готовности к переезду (уточняющий вопрос 4а ниже).

**Вопрос 4а (условный):** Если город не совпадает с локацией офиса — «Готовы ли вы к переезду? За чей счёт?» Стоп: «Не готов к переезду».

---

##### Вопрос 5. Готовность к командировкам

**Текст:** Позиция предполагает командировки (поездки к поставщикам, на таможенные терминалы, к клиентам) — в среднем 2–4 раза в месяц, преимущественно по России. Готовы ли вы к такому режиму?

**Тип:** Одиночный выбор

**Варианты:**
- Да, готов к командировкам в указанном объёме
- Готов к редким командировкам (1 раз в месяц и реже)
- Не готов к командировкам

**Стоп-критерий:** «Не готов к командировкам», если роль это требует. «Готов к редким» — флаг, передать HR.

---

##### Вопрос 6. Опыт работы с таможенным оформлением

**Текст:** Опишите ваш опыт взаимодействия с таможней: вы самостоятельно готовили документы, работали с таможенным брокером или только согласовывали данные?

**Тип:** Открытый ответ

**Стоп-критерий:** Если в анкете указан «опыт международных перевозок более 3 лет», но ответ про таможню — «никогда не сталкивался / передавал брокеру без моего участия» — ставится флаг несоответствия для cross-check.

---

##### Вопрос 7. Знание транспортных накладных и международных конвенций

**Текст:** Какие документы и конвенции вы использовали в практической работе? Отметьте все применимые.

**Тип:** Множественный выбор

**Варианты:**
- CMR-накладная (Конвенция о договоре международной перевозки грузов)
- ЦИМ / CIM (международная железнодорожная накладная)
- СМГС (Соглашение о международном грузовом сообщении)
- Коносамент (Bill of Lading)
- Авиагрузовая накладная (AWB)
- TIR Carnet (Книжка МДП)
- ATA Carnet
- Ни с одним из перечисленных не работал

**Стоп-критерий:** «Ни с одним» при заявленном опыте международных перевозок — автофлаг лжи. Сохраняется для cross-check с тестом.

---

##### Вопрос 8 (ЛОВУШКА №1): Опыт работы с системой LogiTrack PRO X7

**Текст:** Одна из используемых нами TMS-систем — LogiTrack PRO X7. Оцените ваш уровень владения этой системой.

**Варианты:**
- Работал активно (более года)
- Знаком, использовал частично
- Слышал, но не работал

##### Вопрос 2. Расчёт рентабельности маршрута — открытый ответ (3 балла)

**Текст:** Клиент запрашивает доставку FTL Москва — Екатеринбург. Себестоимость перевозчика: 45 000 руб. Ваши операционные расходы на обработку: 3 500 руб. Клиент просит скидку 10% от вашей цены 62 000 руб. Какова будет ваша маржа в рублях и % при предоставлении скидки? Предоставите ли вы скидку?

**Расчёт:**
- Цена со скидкой: 62 000 × 0,9 = 55 800 руб.
- Себестоимость полная: 45 000 + 3 500 = 48 500 руб.
- Маржа: 55 800 – 48 500 = 7 300 руб.
- Маржинальность: 7 300 / 55 800 = 13,1%

**Оценка AI:** 3 балла — верный расчёт + осознанное решение по скидке (с обоснованием: «дам скидку, если клиент обещает объём» или «не дам, так как маржа уже ниже норматива X%»); 2 балла — верный расчёт без стратегии по скидке; 1 балл — методологически верно, арифметическая ошибка; 0 баллов — неверный подход.

**Ловушка 3:** «2–3 поставки в месяц» — это регулярный клиент, потенциально ценный для длительного контракта. Кандидат-эксперт предложит годовой контракт с фиксированными условиями, а не разовую цену.

**Ловушка 4:** Возражение про предоплату 50% vs постоплата у конкурента. Кандидат должен либо объяснить, зачем предоплата (защита от дебиторки, подтверждение серьёзности), либо предложить альтернативу (аккредитив, снижение предоплаты при контракте на объём).

## 12. Технические требования

- [ ] Рандомизация ловушек: пул 5–7 вариантов на роль, случайный выбор для каждой сессии
- [ ] Таймер Этапа 2 проверяется на **сервере**: `submitted_at - started_at <= 30min`
- [ ] Кандидат не может вернуться к предыдущему вопросу в Этапе 2
- [ ] Ответы Этапа 1 передаются AI при оценке всех последующих этапов
- [ ] Feature flag: `ASSESSMENT_SYSTEM_ENABLED` (default: `false`)
- [ ] Просрочка 24h между этапами → статус `expired`

### Дополнительные технические замечания

1. **Рандомизация ловушек:** вопросы-ловушки не должны быть предсказуемы. Рекомендуется иметь пул из 5–7 вариантов ловушек для каждой роли, выбираемых случайно для каждого кандидата.

2. **Таймер:** Этап 2 — строгий таймер 30 минут. Нельзя возвращаться к предыдущим вопросам. Это снижает возможность поиска ответов в интернете.

3. **Последовательность этапов:** обязательно строго последовательная. Нельзя перейти к Этапу 3 без завершения Этапа 2. Промежутки между этапами — максимум 24 часа (контроль целостности данных).

4. **Хранение данных для cross-check:** все ответы Этапа 1 должны быть доступны AI-оценщику при оценке Этапов 2–4 для выполнения cross-check.

5. **Калибровка AI-оценщика:** первые 50 оценённых кандидатов рекомендуется оценивать параллельно с HR-специалистом для калибровки весов и порогов.

6. **Обновление ловушек:** вопросы-ловушки (несуществующие системы, компании, термины) необходимо обновлять каждые 6 месяцев. Если ловушки становятся известны на форумах кандидатов — они теряют эффективность.

7. **Этические оговорки:** кандидаты должны быть уведомлены, что тестирование включает автоматическую проверку, но не должны знать о конкретных механизмах ловушек. Это соответствует стандартной практике психометрических тестирований с L-шкалой.
