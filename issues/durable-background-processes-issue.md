> **Approach:** TDD-first
> **Depends on:** PR #95 (durable-очередь на PostgreSQL)

## 1. Контекст

После PR #95 в проекте появляется durable-очередь на PostgreSQL. Шесть текущих in-memory очередей через `createInMemoryQueue` должны стать durable автоматически за счет замены реализации фабрики.

Эта задача покрывает оставшиеся фоновые процессы, которые не проходят через фабрику и сейчас могут теряться при падении или рестарте процесса.

---

## 2. Цель

Перевести оставшиеся fire-and-forget, in-process и cron фоновые операции на надежную модель выполнения:

- durable-задачи там, где работа должна переживать рестарт;
- идемпотентность обработчиков;
- retry при сбоях;
- журналирование запусков для cron-задач;
- сохранение мультитенантности через `tenantId`;
- без изменения интерфейса очереди из PR #95.

---

## 3. Selection bridge через durable-задачу

### Текущее поведение

`registerSelectionApplicationBridge` в `selection-application-bridge.ts` подписан на in-process событие `application.created` и выполняет `handleApplicationCreatedForSelection` инлайн в обработчике события.

Если процесс падает между emit события и завершением обработчика, авто-создание selection session теряется без следа.

### Нужно сделать

На событие `application.created` мост должен только поставить durable-задачу, например в очередь:

```text
selection.bridge
```

Обработчик durable-задачи уже выполняет `handleApplicationCreatedForSelection`.

### Требования

- Одна selection session на одну application.
- Повторная обработка задачи не должна создавать дубликаты.
- `tenantId` должен передаваться и использоваться явно.
- При рестарте процесса задача должна остаться в очереди и выполниться позже.

## 4. Скоринг публичного отклика через durable-очередь

### Текущее поведение

В `public.routes.ts` скоринг запускается напрямую через fire-and-forget:

```ts
void scoreApplication(...).catch(() => {})
```

При рестарте или падении процесса задача теряется.

### Нужно сделать

Заменить прямой вызов на enqueue в durable-очередь:

```text
application.ai_scoring
```

Использовать ту же очередь и обработчик, которые уже применяются для AI scoring внутри системы.

### Требования

- Не запускать `scoreApplication` напрямую из HTTP route.
- HTTP flow публичного отклика не должен ждать завершения скоринга.
- Ошибка скоринга должна обрабатываться через retry-механику очереди.
- Сохранить текущий UX и response публичной формы.

## 5. Надежность audit log

### Текущее поведение

В `backend/src/http/audit.ts` аудит пишется через `queueMicrotask`.

При крэше процесса часть audit-записей может потеряться. Это compliance-риск.

### Нужно сделать

Сделать запись audit log надежной.

Предпочтительный вариант:

- писать audit log в той же DB-транзакции, что и бизнес-действие.

Если действие сейчас невозможно удобно обернуть общей транзакцией:

- ставить durable-задачу для записи audit log;
- задача должна переживать рестарт;
- запись должна быть идемпотентной.

### Требования

- Не использовать `queueMicrotask` для критичного audit log.
- Не терять audit-записи при падении процесса после успешного бизнес-действия.
- Если бизнес-транзакция откатилась, audit-запись успешного действия не должна появляться как будто действие состоялось.
- Сохранить `tenantId`, actor/user context, entity/action metadata.

## 6. Идемпотентность, retry и журнал запусков cron-задач

### Текущее поведение

В `backend/src/cron.ts` есть 9 cron-задач:

- `probation.reminder`
- `1on1.reminder`
- `okr.quarter_start`
- `review.reminder`
- `analytics.snapshot`
- `signals.compute`
- `data.retention`
- `selection.retention_outcomes`
- `selection.retention_calibration`

Если запуск падает, окно считается пропущенным до следующего расписания.

### Нужно сделать

Добавить надежную модель выполнения cron-задач:

- идемпотентность повторного запуска;
- retry при сбое;
- журнал запусков для наблюдаемости.

### Журнал запусков

Добавить таблицу или модель для cron run log, например:

```text
cron_job_runs
```

Минимальные поля:

- `id`
- `jobName`
- `tenantId` если задача tenant-scoped
- `scheduledFor` или логическое окно запуска
- `startedAt`
- `finishedAt`
- `status`: `running | succeeded | failed`
- `attempt`
- `error`
- timestamps

### Требования

- Повторный запуск одного окна не должен дублировать эффект.
- Упавший запуск должен быть виден в журнале.
- Ошибка должна сохраняться в читаемом виде без секретов.
- Retry должен повторять то же логическое окно, а не ждать следующего расписания.
- Для multi-tenant задач учитывать `tenantId`.
- Не ломать существующее расписание и текущие side effects.

## 8. Acceptance Criteria

- [ ] `application.created` больше не выполняет selection bridge inline, а ставит durable-задачу.
- [ ] Selection bridge задача переживает рестарт и после обработки создает не более одной session на application.
- [ ] Публичный отклик ставит AI scoring в durable-очередь `application.ai_scoring`.
- [ ] В `public.routes.ts` больше нет прямого fire-and-forget вызова `scoreApplication`.
- [ ] Audit log больше не зависит от `queueMicrotask` для критичных записей.
- [ ] Audit log не фиксирует успешное действие, если бизнес-транзакция откатилась.
- [ ] Для cron-задач есть журнал запусков со статусами, временем, attempt и ошибкой.
- [ ] Cron-задачи идемпотентны по логическому окну запуска.
- [ ] Упавшие cron-запуски ретраятся.
- [ ] Все изменения сохраняют `tenantId` там, где задача tenant-scoped.

## 9. Тесты

Добавить или обновить тесты на следующие сценарии:

### Selection bridge

- событие `application.created` ставит durable-задачу;
- после имитации рестарта задача остается доступной;
- повторная обработка не создает duplicate session.

### Public AI scoring

- публичная подача заявки enqueue-ит задачу в `application.ai_scoring`;
- прямой `scoreApplication` из route не вызывается.

### Audit log

- audit-запись не теряется после успешного действия;
- при откате бизнес-транзакции audit не записывает действие как успешное;
- нет зависимости от `queueMicrotask`.

### Cron

- повторный запуск одного окна не дублирует эффект;
- failed run попадает в журнал;
- failed run ретраится;
- successful retry обновляет статус или создает отдельную attempt-запись согласно выбранной модели.
